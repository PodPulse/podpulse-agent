# PodPulse Agent

> ⚠️ **Early development** — We are looking for design partners running Kubernetes in production. [Get in touch](#status)

**AI-powered Kubernetes incident diagnostics.**

PodPulse Agent monitors your Kubernetes cluster in real time, detects production incidents, analyzes root causes using AI (powered by Claude), and opens GitHub Pull Requests with structured fix recommendations — so your team can review and merge, never apply blindly.

The goal is simple: turn Kubernetes incidents into a reviewed, mergeable fix in minutes.

---

## Why PodPulse?

Operating Kubernetes in production often means spending hours investigating:

- OOMKilled pods
- CrashLoopBackOff
- Failed deployments
- Misconfigured resources
- Autoscaling issues

Engineers typically have to manually inspect `kubectl describe`, `kubectl logs`, `kubectl events`, metrics, and Git history.

**PodPulse automates that investigation.**

Instead of debugging manually, PodPulse detects incidents, diagnoses root causes using AI, and opens a PR with a structured recommendation — ready for your team to review.

### Example PR

```markdown
## 🔍 Detection
Pod 'payment-api' in namespace 'payments' has been OOMKilled 4 times in the last 10 minutes.

## 🧠 Root Cause Hypothesis
Container memory limit (256Mi) is too low for the current workload.
Recent batch processing changes likely increased peak memory consumption.

## 🛠 Proposed Fix
Increase memory limit from 256Mi to 512Mi.
Consider adding a VPA in recommendation mode to right-size automatically.

**Confidence score: 0.87**

---
*Generated by PodPulse — review before merging*
```

## How It Works

```
Kubernetes Cluster
       │
       │  Events / Logs / Pod State (read-only)
       ▼
PodPulse Agent (in-cluster, Helm-deployed)
       │
       │  Structured incident report (gRPC)
       ▼
PodPulse Backend
       │
       ▼
AI Diagnostics Engine (Claude Sonnet)
       │
       ▼
GitHub Pull Request
```

The agent runs inside your cluster and observes pod lifecycle events, container statuses, and logs. It sends structured incident data to the PodPulse backend. The agent itself never calls any external AI API — all analysis happens in the backend.

## Features

**Current capabilities (W8):**
- Real-time Kubernetes event and pod status streaming
- AI-powered root cause analysis (Claude Sonnet)
- GitHub PR generation with structured diagnostics

## Installation

```bash
helm repo add podpulse https://podpulse.github.io/podpulse-helm
helm install podpulse-agent podpulse/podpulse-agent \
  --namespace podpulse \
  --create-namespace \
  --set agent.backend.address=<backend-address>:5051 \
  --set agent.backend.apiKey=<your-api-key>
```

Contact us to get an API key: open a [Discussion](https://github.com/PodPulse/podpulse-agent/discussions) or reach out directly.

## Permissions

The agent requires **read-only** access to:

```yaml
apiGroups: [""]
resources:
  - pods
  - pods/log
  - events
  - nodes
  - namespaces
verbs: ["get", "list", "watch"]

apiGroups: ["apps"]
resources:
  - deployments
  - replicasets
verbs: ["get", "list", "watch"]
```

**The agent never modifies cluster resources.**

## Security

PodPulse is designed with security in mind:

- **Read-only** Kubernetes permissions — no write access, ever
- **No cluster modifications** — ever
- **PR-only remediation** — fixes are always proposed as pull requests, never applied directly
- **Secrets never transmitted** — Kubernetes Secrets and environment variable values are never collected
- **Distroless runtime** — minimal image, no shell, runs as nonroot
- **Per-cluster API key** — fine-grained revocation per cluster
- **Optional anonymization** — pod names and namespaces can be anonymized before transmission (future)
